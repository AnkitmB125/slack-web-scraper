module Flightable
  def fly
    puts "I am #{@name}, I am a #{self.class.to_s.downcase}, and I can fly!"
  end
end

class Superhero  
  attr_accessor :ability
  
  include Flightable
  
  def self.fight_crime
    puts "I am #{self}!"
    me = LSMan.new("myself")
    me.ability = Ability.new("coding skills")
    me.announce_ability
  end
  
  def initialize(name)
    @name = name
  end
  
  def announce_ability
    puts "I fight crime with my #{ability.description} ability!"
  end
end

class LSMan < Superhero; end

class Ability
  attr_reader :description

  def initialize(description)
    @description = description
  end
end

superman = Superhero.new('Superman')
superman.fly # => I am Superman, I am a superhero, and I can fly!
LSMan.fight_crime 
# => I am LSMan!
# => I fight crime with my coding skills ability!

Hey @riaz, so I might not have a totally thorough technical answer, but I’m going to give this my best shot. This is really interesting code! I’d never thought to expose a private method to a public method and then use that public method for comparison.Essentially though, you’ve created a nice workaround to private methods not being able to be used for comparison. The original getter method is still private, but private methods can be called from public methods within the class, so when you call current_value, it returns the value of instance variable @value. The getter method is still private, however it doesn’t really matter once you use the public method current_value , as it is a public method, and therefore can be used for comparison. You’ve essentially made the private method value public with current_value, if that makes sense. It still stands though that private methods can’t be used for comparisons, because if you used the private value reader in your > method, it would raise an error.
def >(other_house)

As far as your question about collaborators, no, as @Aryan Binazir pointed out, house2 is not a collaborator because it is not a part of house1 ’s state.
